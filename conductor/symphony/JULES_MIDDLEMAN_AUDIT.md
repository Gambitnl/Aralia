# Jules Middleman Completion Audit

This audit tracks the active goal of turning Symphony into Aralia's dashboard-first foreman console and Jules delegation middleman.
The durable scope reference is `conductor/symphony/docs/JULES_MIDDLEMAN_OPERATING_SPEC.md`; the goal should point to that spec and this audit instead of growing into a static prompt that loops forever.

It is intentionally evidence-based: a requirement is only marked proven when current code, verifier coverage, rendered artifacts, or live state directly supports it. Fixture tests are useful contract evidence, but they do not replace a real Jules/GitHub/Linear end-to-end run.

This file is also the live Markdown status ledger for the active Symphony goal. Each major task area must keep its goalpost, current status, achieved/not-achieved state, blocker, remaining proof, and evidence path visible here instead of hiding status in conversation memory.

## Current Verdict

**Not complete yet.**

The local dashboard/API contract is broadly implemented and covered by the `verify:jules-contract` suite. The old ever-growing thread goal has been deleted, but the bigger goal remains the dashboard-first middleman outcome: Symphony should coordinate local Codex, Jules, GitHub, Linear, Scout/Core, and local repository sync from one trustworthy operator surface. The current stage inside that larger goal is **dashboard foreman-console focus, dynamic worker-mode proof, and default-off dispatch control**. A full completion claim still needs real external proof: a dashboard-started task must travel through Linear, the existing Jules orchestrator, a GitHub PR, Scout/Core readiness, and local sync.

The newest dashboard-focus finding is that the main dashboard is housing too many first-class surfaces at once: header controls, run totals, worker roster, task intake, existing-PR watch, drafts, handoffs, routing, nudge ledger/scheduler, Git preflight, Git disposition/review/resolution/sync-plan packets, middleman path, kickoff sequence, handoff readiness/pass-path, Linear and manifest previews, Jules launch/session controls, PR readiness, check artifacts, feedback classification, Scout/Core readiness, conflict watch, local sync, usage, approvals, running/retrying workers, and activity. The spec now records that inventory and names the next goal as a current-boundary-focused foreman console rather than a wall of equally loud cards.

Dynamic Codex reasoning by complexity is now locally implemented as a worker-mode packet. Current code routes tasks to `operator_only`, `local_fast`, `local_careful`, `jules_task`, `jules_plan`, or `observe_wait`, recommends a model/reasoning effort from task complexity and boundary state, and keeps explicit `codex.model` / `codex.reasoning_effort` as the launch-time override.

Symphony dispatch is now backend-gated and default-off on process startup. The dashboard can stay open for state inspection, task intake, Git preflight, and proof capture without polling trackers, claiming issues, cleaning workspaces, or starting Codex workers until the operator enables `/api/v1/dispatch-control`.

## Browser Constraint

Live dashboard inspection must use the built-in Codex app browser feature so the user can follow along. Do not use an external Chrome/Chromium browser window for interactive progress checks unless the user explicitly changes this constraint.

Headless Playwright/Chromium verifier scripts may still exist as repeatable contract tests, but they are not a substitute for follow-along live inspection in the Codex app browser.

The compact `/proof` board is the current follow-along fallback for proof capture. It is server-rendered, script-free, and backed by the same task-intake snapshot as `/api/v1/task-drafts`, so it avoids the full dashboard's screenshot-size problem without creating a second orchestration source of truth.
