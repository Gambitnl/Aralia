| WSS-004 | **[REMEDIATED 2026-06-02] Silent flat-world fallback.** When `MapData` reaches migration with neither `worldData` nor `azgaarWorld`, `migrateMapDataToWorldDataV2` backfills constant `heights = 30` (temp 15, moisture 20, templateId `'unknown'`), so `runWorldSim` emits a featureless flat heightfield. **Verified live path** (not hypothetical): `generateMap` falls back to `generateLegacyMap` on any Azgaar-generation throw, and `generateLegacyMap` builds only biome `tiles` — it never sets `azgaarWorld`/`worldData`. | WSS-005 | **2D↔3D feature divergence at the source.** The 2D atlas draws Azgaar's own `azgaarWorld.rivers`, but the 3D world's rivers/sites/roads are **re-derived** by `runWorldSim` from the heightfield (`traceRivers`, `placeSites`, `generateRoads`) — a different algorithm. The two surfaces can disagree on where rivers/towns/roads sit for the same world. | `src/services/azgaarDerivedMapService.ts:502-509` (stores `azgaarWorld.rivers`); `src/services/worldSim/index.ts:63-65` (`traceRivers`/`placeSites` regenerate independently) | The `WorldData` contract and the generation pipeline that fills it are owned here; deciding the single source of truth is a generation concern. | Choose one source of truth: either feed Azgaar's `rivers` (and site hints) into `WorldData.rivers`/`sites`, or render the 2D atlas from `WorldData` output, so both surfaces agree. | WSS-006 | **Biome-fallback climate is still flat.** The WSS-004 remediation derives real *heights* from biomes, but the no-Azgaar path still backfills `temperatures = 15` and `moisture = 20` as uniform constants. So on the biome-derived path the heightfield has relief while the climate fields are a single value everywhere — any downstream consumer keying off temperature/moisture (snow lines, climate tinting, future biome refinement) sees a flat climate that contradicts the varied terrain. | `src/state/migrations/worldDataMigration.ts` (`temperatures`/`moisture` still `new Array(...).fill(15/20)` on the `!az?.heights` branch); `src/services/worldSim/heightFromBiomes.ts` (derives heights only) | Same generation-fidelity concern as WSS-004, same file/path; owned here. | Derive temperature/moisture from biome climate/moisture bands (or from latitude+height) deterministically, mirroring `heightFromBiomes`; assert variance + seed-determinism. |
| WSS-007 | **Biome-derived heightfield has no spatial smoothing → cliff seams.** `heightFromBiomes` assigns each cell an independent band height + jitter with no neighbour blending, so adjacent cells in different bands jump abruptly (e.g. low ~32 next to high ~74), producing vertical walls at every biome border in 3D. The Azgaar path smooths heights (`azgaarDerivedMapService` blur passes); the fallback does not, so fallback worlds look terraced/blocky rather than sloped. | `src/services/worldSim/heightFromBiomes.ts` (per-cell independent assignment, no neighbour averaging); contrast `src/services/azgaarDerivedMapService.ts:271-275` (blur pass) | The derivation is owned here and is the new fallback fidelity surface introduced by WSS-004. | Add a deterministic smoothing/relaxation pass (e.g. N neighbour-average iterations seeded identically) after band assignment; assert borders slope and output stays seed-deterministic. |

## Open Questions

- Should `worldSim` expose a secondary fast path for legacy-only or low-end devices to avoid synchronous regeneration at load?
- Should `worldDataMigration` include checksum or generation metadata (`seed`,`templateId`,`inputHash`) to support trust-safe rehydration?
- What is the acceptable fallback behavior when a load-time regenerated world snapshot diverges from a display cache that assumes older feature density?
