# Crime UI North Star

Status: active
Last updated: 2026-05-31

## Purpose And Scope

This project tracks UI-facing Crime features for one feature area only: Thieves Guild, safehouse, and heist interface wiring.

This is not the full crime domain implementation. Core logic, math, and bounty lifecycle remain in `docs/projects/crime/`.

## Current State (verified by scan, docs-only update)

- The Crime UI is implemented and wired through the modal manager.
- Evidence shows a partial but real flow for:
  - Thieves Guild membership and jobs (`src/components/Crime/ThievesGuild/ThievesGuildInterface.tsx`)
  - Fence item sale flow (`src/components/Crime/ThievesGuild/FenceInterface.tsx`)
  - Heist planning modal and phase advancement (`src/components/Crime/ThievesGuild/HeistPlanningModal.tsx`)
  - Safehouse service UI (`src/components/Crime/ThievesGuild/ThievesGuildSafehouse.tsx`)
- State wiring exists for guild/heist actions in:
  - `src/state/actionTypes.ts` (thieves/ heist action types)
  - `src/state/reducers/crimeReducer.ts` (COMMIT_CRIME, JOIN_GUILD, job/service/heist actions)
  - `src/state/reducers/uiReducer.ts` (isThievesGuildVisible, isThievesGuildSafehouseVisible)
- Modal entry/close is routed in:
  - `src/components/layout/GameModals.tsx`

## File Map

- `src/components/Crime/ThievesGuild/ThievesGuildInterface.tsx`  
  Main UI: membership rank gate, job list, active jobs, service list.
- `src/components/Crime/ThievesGuild/FenceInterface.tsx`

- `src/state/reducers/crimeReducer.ts`  
  Business logic transitions for thieves guild jobs/services/heists.
- `src/state/reducers/uiReducer.ts`  
  Modal visibility toggles for thieves guild and safehouse.
- `src/systems/crime/ThievesGuildSystem.ts`  
  Job and service generation, including service IDs for UI to show/use.
- `src/systems/crime/HeistManager.ts`  
  Heist phase and action mechanics used by planning UI.
- `src/types/crime/index.ts`  
  Heist and guild types for action contracts.

## Implemented State

- Modal integration is present and functional enough to open/close UI entry points.
- Guild membership is used as a lazy default in reducer when state is missing (`rank 0` path), so membership actions have a fallback.
- Guild jobs are generated, set to state, and accept/complete/abandon actions exist.
- The heist plan has planning mode in reducer/system and is rendered by the planning modal.

## Integrations And Ownership

- This project is the UI boundary for crime systems.
- Core Crime mechanics, bounty models, and expiry/heat behavior are owned by:
  - `docs/projects/crime/NORTH_STAR.md`
  - `docs/projects/crime/TRACKER.md`
  - `docs/projects/crime/GAPS.md`
- This project should not be used to absorb general crime-system logic; it should only raise or link flow-level gaps that block UI.

## Gaps And Uncertainties

- No direct suspect/report flow is present in the `src/components/Crime` or `src/systems/crime` scan area.
- Fence sales currently dispatch generic `SELL_ITEM`, with no dedicated crime-specific transaction action and limited heat/exposure side-effects in UI wiring.
- `HeistPlanningModal` uses a local interface cast for approaches/intel, indicating plan shape assumptions still cross boundaries.
- Safehouse services are hardcoded in one component and also generated by system, which risks temporary divergence.
